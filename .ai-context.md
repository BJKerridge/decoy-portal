# SYSTEM CONTEXT: DECOY-Portal

This file serves as the source of truth for the architecture, tech stack, and conventions of the DECOY-Portal project. Refer to this context before proposing modifications or creating code artifacts.

## Project Overview
DECOY-Portal is a multi-component tactical application built to support EVE Online alliance tracking, sovereign status metrics, and operations dashboard management. It functions as a monorepo containing distinct decoupled components that interact over an internal Docker network.

---

## Component Registry

| Component Name | Directory | Tech Stack | Responsibility |
| :--- | :--- | :--- | :--- |
| **DECOY-UI** | `/decoy-ui` | Next.js, TypeScript, TailwindCSS | User-facing dashboard interface. Operates cleanly with live or stubbed mock data for rapid frontend isolation testing. |
| **DECOY-Orch** | `/decoy-orch` | Next.js / Node.js API | API orchestration layer. Exposes public internal endpoints to the UI, fetching real-time data from either `DECOY-DB` or proxies to the EVE Online ESI. |
| **DECOY-DB** | Managed | PostgreSQL (v16) | Target transactional system database. Houses persistent data schemas, active access tokens, and cached configurations. |
| **DECOY-DataLayer**| `/decoy-datalayer`| Java 21, Maven/Gradle | Heavy background ingestion engine. Handles async cron loops, credential/token rotation, and streams data from EVE ESI or zKillboard. |
| **DECOY-OTEL** | `/decoy-otel` | OpenTelemetry Collector | Intermediary collection layer routing traces, logs, and performance metrics out to a monitoring endpoint. |

---

## Architecture Flow & Boundaries

1. **Frontend Request Loop:** `DECOY-UI` $\rightarrow$ calls endpoints on $\rightarrow$ `DECOY-Orch`.
2. **Orchestration Boundary:** `DECOY-Orch` may read from `DECOY-DB` directly or pull instant dynamic payloads. It **never** executes write-intensive cron or sync routines.
3. **Background Sync Loop:** `DECOY-DataLayer` continuously updates `DECOY-DB` and hits external APIs asynchronously.

---

## Development Constraints & Conventions

* **Decoupled Isolation:** Changes to `/decoy-ui` must not break or tightly depend on structural files within `/decoy-datalayer`. Components must run individually inside their respective directories or concurrently via the root orchestration script.
* **Environment Variables:**
  * Do not hardcode database secrets or API access codes.
  * Use the root `docker-compose.yml` configuration as the target reference for system networking names (e.g., connect to database using host `decoy-db`).

---

  ## Telemetry & Logging Standards

* **Application Logs:** Both `DECOY-DataLayer` (Java) and `DECOY-Orch` (Node/Next) write logs primarily via standard output (`System.out.println` or standard logger suites). 
* **Log Shipping:** The `decoy-promtail` agent monitors the local Docker socket (`/var/run/docker.sock`) to ship these logs directly to `decoy-loki`.
* **Telemetry Queries:** Logs are visualized in Grafana (`decoy-monitoring`) on port `4000` using LogQL target selectors (e.g., `{container="decoy-datalayer"}`).
* **Metrics & Traces:** Complex traces are explicitly targeted to the `decoy-otel` collector via OTLP endpoints on ports `4317` (gRPC) and `4318` (HTTP).

## Deployment Modes & Composition Layout

The orchestration layers are divided explicitly using Docker Compose configuration merging to separate development utilities from hardened target environments:

* **Base Configuration (`docker-compose.yml`):** Maps core container services, build scopes, basic dependencies, and foundational network bridges.
* **Development Sandbox (`docker-compose.override.yml`):** Automatically loaded locally. Exposes native container host ports (`5432`, `4000`, `3001`, `3000`), mounts local development directory state volumes, and appends development triage UIs (`decoy-adminer`, `decoy-monitoring`, `decoy-loki`, `decoy-promtail`).
* **Production Build Strategy (`docker-compose.prod.yml`):** Tears down public host port mapping rules for internal layers, strips local monitoring blocks, attaches permanent infrastructure volume directories, and implements rigorous container restart constraints (`restart: always`).

---

## Infrastructure & Secrets Management

* **Configuration:** All environment-specific variables, database names, and master authentication credentials live in a local, uncommitted `.env` file in the project root.
* **Database Initialization:** Database schemas, tables (like `test_table`), and initial constraints are loaded idempotently via SQL scripts placed in `./decoy-db/init/` which map to the container's `/docker-entrypoint-initdb.d/` lifecycle hook. Manual table creation is discouraged.
